An recursive LLM that lives inside a bash shell. It thinks by writing code, runs it, sees the output, and iterates until it has an answer. Give it a task — it figures out the rest.

## How it works

shelllm runs a loop:

1. Sends your context to Claude with a system prompt that says "write bash code"
2. Claude responds with a `\`\`\`bash` code block
3. shelllm executes the code (in Docker if available, locally otherwise)
4. Output streams back to Claude as the next message
5. Repeat until the code sets `FINAL="answer"` or hits max iterations

The LLM has full shell access. It can curl APIs, parse data with jq, write Python scripts, call itself recursively, install packages — whatever it takes to solve the task.

## Quick start

```bash
# Prerequisites: bash, jq, curl
# Optional: Docker (auto-detected, used for sandboxing)

# Set your API key
export ANTHROPIC_API_KEY="sk-ant-..."

# Run it
./shelllm what is the mass of jupiter in kilograms

# Pipe data in
cat dataset.csv | ./shelllm summarize this data and find outliers

# Pass files
./shelllm -f paper.pdf -f notes.txt compare these documents

# Quotes are optional for inline context
./shelllm research the latest advances in protein folding and write a summary
```

## What you see

shelllm streams everything live. Commands show in cyan, output in dim:

```

▶ Executing bash (12 lines):
    curl -s "https://api.example.com/data" | jq '.results'
    ...

  $ jq '.results'
  │ {"count": 42, "items": [...]}
  $ FINAL="Found 42 results"
  Exit 0

▶ Final answer received
Found 42 results
```

## Docker sandboxing

If Docker is running, shelllm automatically executes code inside a container. This means:

- The LLM can't accidentally modify your host system
- It can install packages (`apt-get install`) without affecting your machine
- The container comes with bash, python3, jq, and curl pre-installed
- Your workspace directory is mounted in, so files persist across iterations

```bash
# Auto-detected (default)
./shelllm do something risky

# Force local execution
./shelllm --no-docker do something risky

# Use a different base image
./shelllm --docker-image python:3.12-slim write a flask app
```

Docker detection works like this:
- If Docker daemon is running and `--no-docker` isn't set → uses Docker
- If already inside a Docker container (e.g. CI) → runs locally (no nesting)
- If Docker isn't installed → runs locally

## Context

There are three ways to pass context:

**Inline** — everything after the flags is your context (quotes optional):

```bash
./shelllm explain why the sky is blue
./shelllm "explain why the sky is blue"  # same thing
```

**Files** — use `-f` one or more times. Contents become `$FILE_CONTEXT1`, `$FILE_CONTEXT2`, etc:

```bash
./shelllm -f data.json analyze this
./shelllm -f chapter1.txt -f chapter2.txt compare these chapters
```

**Stdin** — piped data becomes `$CONTEXT`:

```bash
curl https://example.com | ./shelllm what is this page about
git diff | ./shelllm review this diff for bugs
```

These can be combined:

```bash
cat errors.log | ./shelllm -f config.yaml diagnose why the service is failing
```

## Completion signals

The LLM signals it has an answer by setting one of these in its bash code:

```bash
# Short answer
FINAL="The answer is 42"

# Long answer from a file
echo "detailed report..." > report.txt
FINAL_FILE=report.txt
```

shelllm captures the value and prints it to stdout. Everything else (progress, commands, debug) goes to stderr, so you can pipe the final answer:

```bash
./shelllm -f data.csv compute the average > result.txt
```

## Recursive calls

Code generated by shelllm can call shelllm itself:

**`shelllm query "prompt"`** — single-shot LLM call. Fast, cheap, no iteration loop. Good for classification, extraction, simple reasoning:

```bash
# Inside generated code
category=$(shelllm query "classify this text as positive/negative/neutral: $text")
```

**`shelllm query --fork-history "prompt"`** — forks the current conversation history and workspace into a new sub-loop. The sub-agent gets full context of what happened so far, plus its own independent copy of all workspace files:

```bash
# Inside generated code
result=$(shelllm query --fork-history "now take the data we found and build a visualization")
```

Fork-history is useful when a subtask needs multiple iterations and access to the current state but shouldn't pollute the parent's workspace.

Recursion depth is capped by `--max-depth` (default 3) to prevent runaway costs.

## Workspace

Each shelllm run gets a workspace directory (a tmpdir by default) where:

- All generated code executes (`cd $SHELLLM_WORKSPACE`)
- Files created by the agent persist across iterations
- `.final` holds the completion answer
- `.messages.json` holds conversation history (used by `--fork-history`)
- `.output` is used transiently during execution

By default, the workspace is cleaned up when the run ends. Use `--workspace` to keep it:

```bash
./shelllm --workspace ./my-run research something complex

# After the run (or interruption), inspect what happened:
ls ./my-run/
# .final  .messages.json  notes.txt  data.json  ...
```

With Docker, the workspace is bind-mounted into the container at the same path, so files written inside the container appear on the host and vice versa.

## Options

All configuration is available as both CLI flags and environment variables. Flags take precedence.

| Flag | Env var | Default | Description |
|------|---------|---------|-------------|
| `--model` | `SHELLLM_MODEL` | `claude-opus-4-6` | Claude model to use |
| `--max-iterations` | `SHELLLM_MAX_ITERATIONS` | `25` | Max loop iterations before giving up |
| `--max-tokens` | `SHELLLM_MAX_TOKENS` | `16000` | Max tokens per API response |
| `--thinking-budget` | `SHELLLM_THINKING_BUDGET` | `10000` | Extended thinking token budget |
| `--max-depth` | `SHELLLM_MAX_DEPTH` | `3` | Max recursion depth for `--fork-history` |
| `--workspace DIR` | — | tmpdir | Workspace directory (preserved if specified) |
| `--no-docker` | `SHELLLM_NO_DOCKER=1` | off | Disable Docker sandboxing |
| `--docker-image` | `SHELLLM_DOCKER_IMAGE` | `ubuntu:latest` | Docker image to use |
| `-f FILE` | — | — | Add file context (repeatable) |
| `-v, --verbose` | — | off | Show debug output |

You can also put settings in a `.env` file in the working directory:

```bash
ANTHROPIC_API_KEY=sk-ant-...
SHELLLM_MODEL=claude-sonnet-4-5-20250929
SHELLLM_MAX_ITERATIONS=10
```

## Examples

```bash
# Research and summarize
./shelllm research the latest SpaceX launches and summarize in a table

# Process data

# Generate code
./shelllm write a python script that converts markdown to html

./shelllm --model claude-sonnet-4-5-20250929 what is 2+2

# Preserve workspace for debugging
./shelllm --workspace /tmp/debug -v why does this segfault

# Skip Docker for speed
./shelllm --no-docker --max-iterations 1 say hello
```

## How the LLM sees it

The system prompt tells Claude it's operating inside a bash shell. It knows about:

- Standard shell tools (grep, awk, sed, jq, curl, python3, etc.)
- `shelllm query` for recursive LLM calls
- `FINAL` / `FINAL_FILE` for signaling completion
- Context env vars (`$CONTEXT`, `$FILE_CONTEXT1`, etc.)
- Whether it's in Docker (and what tools are available)
- The shared workspace directory

The LLM generates one bash code block per response. The code runs with `set -e` (exit on error). After each execution, the LLM sees the stdout/stderr output and decides what to do next.

## Architecture

```
shelllm (host)
├── main()           Parse flags, dispatch
├── run_loop()       The core iteration loop
│   ├── call_claude()      Send messages, get response
│   ├── extract_code()     Pull bash from markdown
│   ├── execute_code()     Run in Docker or locally
│   │   ├── docker exec    (if Docker available)
│   │   └── bash -e -c     (if local)
│   └── check .final       Done? Output answer. No? Loop.
├── cmd_query()      Recursive call handler
│   ├── simple       Single-shot API call
│   └── fork         Clone workspace + history, run sub-loop
└── Docker lifecycle
    ├── should_use_docker()  Auto-detect
    ├── docker_setup()       Create container, install tools
    └── docker_teardown()    Cleanup on exit
```

Single file. No dependencies beyond bash, jq, and curl. ~820 lines.
